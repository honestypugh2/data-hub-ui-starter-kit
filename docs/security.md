# Security Model — Starter Kit UI (Phase 1)

## Threat Model Summary

The starter kit UI handles organization user credentials and image data. The security design follows a zero-trust architecture (ZTA) with defense in depth across authentication, transport, storage, and API layers.

---

## Authentication & Authorization

### Entra ID (Azure AD) — Frontend

| Property | Value |
|---|---|
| Library | `@azure/msal-browser` v5, `@azure/msal-react` v5 |
| Flow | Authorization Code with PKCE (popup) |
| Token storage | `sessionStorage` (cleared on tab close) |
| MFA | Enforced by tenant Conditional Access policy |
| Scopes | `api://{backend-client-id}/access_as_user` |

- The SPA obtains an access token via `acquireTokenSilent` (with interactive fallback).
- Tokens are never sent to third parties — only to the registered Azure Functions API endpoint.

### JWT Validation — API

| Property | Value |
|---|---|
| Library | `python-jose` |
| Algorithm | RS256 |
| Key source | OIDC discovery (`/.well-known/openid-configuration` → `jwks_uri`) |
| Validated claims | `aud` (must match backend client ID), `iss` (must match authority/v2.0) |
| Key caching | JWKS response cached in-process after first fetch |

Implementation: [app/api/shared/auth.py](../app/api/shared/auth.py)

Every endpoint calls `validate_token(req)` before any business logic. Failure returns `401` or `503` (if OIDC keys unreachable).

### Agency Isolation

The user's agency is derived from the JWT `department` claim (fallback: `agency` claim, default: `"default"`). All data operations scope to the caller's agency:

- Metadata path: `ui-metadata/{agency}/{upload_id}.json`
- List endpoint returns only records under the caller's agency prefix.

This prevents cross-agency data access without explicit RBAC roles.

## Transport Security

| Path | Protection |
|---|---|
| Browser → Azure Functions API | HTTPS (TLS 1.2+, enforced by Azure Functions host) |
| Browser → Azure Blob Storage (SAS upload) | HTTPS (TLS 1.2+, enforced by storage account) |
| Azure Functions → Blob Storage | Private endpoint (ZTA network) or HTTPS with managed identity |
| Azure Functions → Entra ID (OIDC keys) | HTTPS |

No HTTP (plaintext) traffic is permitted in the deployed environment.

## SAS Token Security

### Write SAS (upload)

| Property | Value |
|---|---|
| Generated by | `shared/sas.py → generate_write_sas_url()` |
| Key type | User delegation key (no storage account keys involved) |
| Permissions | `write`, `create` only |
| Scope | Single blob (`bronze/{upload_id}_{filename}`) |
| Expiry | 15 minutes |
| Issued after | JWT validation succeeds |

### Read SAS (preview)

| Property | Value |
|---|---|
| Generated by | `shared/sas.py → generate_read_sas_url()` |
| Key type | User delegation key |
| Permissions | `read` only |
| Scope | Single blob |
| Expiry | 30 minutes |

### Controls

- SAS URLs are **blob-scoped** — they cannot list or access other blobs.
- SAS URLs are **time-limited** — unusable after expiry.
- **User delegation keys** are used instead of storage account keys, enabling revocation via Entra ID.
- The API never exposes storage account keys or connection strings to the client.

## Input Validation

### Client-Side (UploadPage.tsx)

| Check | Rule |
|---|---|
| File type | `image/jpeg` or `image/png` only |
| File size | ≤ 20 MB |
| Feedback | Immediate error message, upload button disabled |

### Server-Side (upload_initiate/init.py)

| Check | Rule | Response |
|---|---|---|
| JWT present | `Authorization: Bearer <token>` required | 401 |
| JWT valid | RS256 signature, audience, issuer | 401 |
| User not null | Explicit `None` guard after `validate_token` | 401 |
| JSON body | Valid JSON required | 400 |
| Filename | Non-empty string | 400 |
| Extension | Must be in `{jpg, jpeg, png}` | 400 |
| Content-Type | Must be in `{image/jpeg, image/png}` | 400 |
| Size | Integer, > 0, ≤ `MAX_UPLOAD_SIZE_MB` × 1 MiB | 400 |
| Filename sanitization | `re.sub(r"[^a-zA-Z0-9_\-]", "_", name)` — strips path traversal, special chars | — |

The server **never trusts the client** — all validations are repeated server-side.

## Blob Storage Security

### Container Access

| Container | Access Level |
|---|---|
| `bronze` | Private (no anonymous access) |
| `gold` | Private |
| `ui-metadata` | Private |

All containers require authentication (SAS token or managed identity). There is no public/anonymous blob access.

### Credential Hierarchy

1. **Deployed**: `DefaultAzureCredential` → system-assigned managed identity (no secrets in config).
2. **Local dev**: `AZURE_STORAGE_CONNECTION_STRING` (for developer convenience only; never deployed).

The storage module (`shared/storage.py`) prefers connection string if set, otherwise falls back to managed identity.

---

## Infrastructure Security (ZTA)

| Control | Implementation |
|---|---|
| Network isolation | Private endpoints for storage and function app |
| No public blob access | Storage account `publicNetworkAccess: Disabled` (production) |
| Admin access | Azure Bastion VM — no direct SSH/RDP exposure |
| Private DNS | Azure Private DNS zones for `*.blob.core.windows.net` |
| Firewall | NSG rules restrict inbound to necessary ports only |

---

## Secrets Management

| Secret | Storage | Access |
|---|---|---|
| Entra ID client IDs | Environment variables / App Configuration | Not secret (public client) |
| Entra ID tenant ID | Environment variables / App Configuration | Not secret |
| Storage connection string | `local.settings.json` (local only) | Never in source control, never deployed |
| Storage account keys | Not used by application | Managed identity replaces key-based auth |
| JWT signing keys | Fetched at runtime via OIDC discovery | Public keys (RS256) |

No secrets are committed to source control. The `.gitignore` excludes `local.settings.json` and `.env` files.

## OWASP Top 10 Alignment

| # | Risk | Mitigation |
|---|---|---|
| A01 Broken Access Control | JWT validation on every endpoint; agency-scoped metadata; blob-scoped SAS | 
| A02 Cryptographic Failures | TLS 1.2+; tokens in sessionStorage (not localStorage); user delegation keys | 
| A03 Injection | Filename sanitization (regex whitelist); no SQL/NoSQL; JSON blob storage | 
| A04 Insecure Design | Two-step SAS upload (API never handles file bytes); least-privilege SAS permissions | 
| A05 Security Misconfiguration | Private containers; no anonymous access; managed identity; private endpoints | 
| A06 Vulnerable Components | Vite build with 0 npm vulnerabilities; MSAL v5; no legacy CRA dependencies | 
| A07 Auth Failures | Entra ID with MFA; RS256 JWT with audience/issuer checks; OIDC key rotation via discovery | 
| A08 Data Integrity Failures | SAS URLs are blob-scoped and time-limited; upload metadata created server-side | 
| A09 Logging & Monitoring | Python `logging` module; Azure Functions host logging; Application Insights (infra) | 
| A10 SSRF | API does not make requests based on user-supplied URLs; OIDC endpoint is hardcoded to Entra ID |

## Incident Response Notes

- **Compromised SAS URL**: Expires in 15 minutes (write) or 30 minutes (read). Revoke the user delegation key at the storage account to invalidate all outstanding SAS URLs immediately.
- **Compromised JWT**: Revoke the user's session in Entra ID; tokens are short-lived (default 1 hour). `sessionStorage` ensures tokens are lost on tab close.
- **Compromised storage account key**: Not applicable — the application uses managed identity, not account keys.
